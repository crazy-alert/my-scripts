🔐 Инструкция по настройке безопасности Ubuntu

Эта инструкция поможет вам защитить сервер Ubuntu: настроить SSH только по ключам, установить и настроить fail2ban, а также межсетевой экран ufw с открытием только нужных портов (80, 443, 8448).

⚠️ Подготовка и важные предупреждения

Внимание! Перед выполнением команд убедитесь, что:
  • У вас есть доступ по SSH с использованием ключа (публичный ключ добавлен в ~/.ssh/authorized_keys). После отключения парольного входа вы сможете зайти только по ключу.
  • Вы имеете root-доступ или используете sudo для всех команд.
  • Вы понимаете, что открываются порты 80, 443 и 8448. Если вам нужны другие порты, измените команды соответственно.

!!! Шаг 0 — добавления сертификата на сервер

Без этого никак

XXX.XXX.XXX.XXX - ip сервера

C:\Users\user\.ssh\id_ed25519.pub - действительный путь к публичному сертификату на вашей машине

type C:\Users\user\.ssh\id_ed25519.pub | ssh root@XXX.XXX.XXX.XXX "cat >> .ssh/authorized_keys"

📦 Шаг 1 — Обновление пакетов

Рекомендуется обновить список пакетов перед началом установки:

sudo apt update

🔥 Шаг 2 — Установка и настройка ufw (межсетевой экран)

Устанавливаем ufw (если не установлен), задаём политики по умолчанию и открываем порты 80, 443, 8448.

# Установка ufw (если отсутствует)
sudo apt install ufw -y

# Политики по умолчанию: запрет входящих, разрешение исходящих
sudo ufw default deny incoming
sudo ufw default allow outgoing

# Разрешаем необходимые порты
sudo ufw allow 22/tcp comment 'ssh'
sudo ufw allow 80/tcp comment 'HTTP'
sudo ufw allow 443/tcp comment 'HTTPS'
sudo ufw allow 8448/tcp comment 'Matrix (или другой сервис)'

# Включаем ufw
sudo ufw --force enable

# Проверяем статус
sudo ufw status verbose

🛡️ Шаг 3 — Установка и настройка fail2ban

Устанавливаем fail2ban, создаём базовую конфигурацию для защиты SSH (и других сервисов при необходимости).

# Установка fail2ban
sudo apt install fail2ban -y

# Создаём локальный конфигурационный файл, если его нет
sudo tee /etc/fail2ban/jail.local > /dev/null <<'EOF'
[DEFAULT]
# Блокировка на 10 минут
bantime = 600
# Период поиска неудачных попыток (10 минут)
findtime = 600
# Игнорировать локальные адреса
ignoreip = 127.0.0.1/8 ::1

[sshd]
enabled = true
port    = ssh
logpath = %(sshd_log)s
backend = %(sshd_backend)s
EOF

# Перезапускаем fail2ban и добавляем в автозагрузку
sudo systemctl restart fail2ban
sudo systemctl enable fail2ban

# Проверяем статус SSH jail
sudo fail2ban-client status sshd

🔑 Шаг 4 — Отключение входа по паролю в SSH (только ключи)

Редактируем конфигурацию SSH-сервера: запрещаем аутентификацию по паролю, оставляем только ключи. Перед выполнением убедитесь, что у вас есть рабочий ключ!

Важно: После выполнения этих команд и перезапуска SSH войти по паролю будет невозможно. Если у вас нет ключа — вы потеряете доступ. 
# Создаём резервную копию текущей конфигурации SSH
sudo cp /etc/ssh/sshd_config /etc/ssh/sshd_config.backup.$(date +%Y%m%d%H%M%S)

# Отключаем парольную аутентификацию
sudo sed -i 's/^#*\(PasswordAuthentication\) .*/\1 no/' /etc/ssh/sshd_config
sudo sed -i 's/^#*\(ChallengeResponseAuthentication\) .*/\1 no/' /etc/ssh/sshd_config
# Включаем аутентификацию по ключам (на всякий случай)
sudo sed -i 's/^#*\(PubkeyAuthentication\) .*/\1 yes/' /etc/ssh/sshd_config
# Убеждаемся, что PAM включён (обычно да)
sudo grep UsePAM /etc/ssh/sshd_config

# Проверяем конфигурацию на ошибки
sudo sshd -t

# Если проверка прошла успешно, перезапускаем SSH
sudo systemctl restart ssh

✅ Шаг 5 — Завершение и проверка

Проверьте состояние служб и убедитесь, что всё работает как ожидалось.

# Статус ufw
sudo ufw status verbose

# Статус fail2ban (общий и по SSH)
sudo fail2ban-client status
sudo fail2ban-client status sshd

# Проверка открытых портов (должны быть видны 80, 443, 8448)
sudo ss -tulpn | grep -E ':80|:443|:8448'

Теперь можно открыть новое окно терминала и попробовать подключиться по SSH с использованием ключа. Если всё работает — отлично!

🐳 Дополнительно: защита Docker-контейнеров с fail2ban

Если на сервере работают сервисы в Docker (например, веб-приложения на портах 80/443), стандартный fail2ban на хосте не сможет их защитить без дополнительной настройки. Проблема в том, что:

  • Логи контейнеров обычно не видны fail2ban на хосте (или требуют монтирования).
  • Docker использует свою цепочку iptables DOCKER-USER, а стандартные действия fail2ban добавляют правила в цепочку INPUT (только для трафика к хосту).

Чтобы защитить, например, веб-сервер в контейнере:

  1. Создайте отдельную «тюрьму» (jail) для вашего сервиса в /etc/fail2ban/jail.local.
  2. Укажите правильный путь к логам (пробросьте логи контейнера на хост или используйте драйвер логирования).
  3. Обязательно добавьте параметр chain = DOCKER-USER в действие, чтобы блокировка применялась к трафику, идущему в контейнеры.

Пример настройки для гипотетического веб-приложения в контейнере (добавьте в /etc/fail2ban/jail.local):

[docker-webapp]
enabled  = true
port     = http,https
filter   = webapp-filter   # имя фильтра (например, nginx или свой)
logpath  = /var/log/docker/webapp/*.log   # путь к логам контейнера на хосте
action   = iptables-allports[name=webapp, chain=DOCKER-USER]
maxretry = 3
bantime  = 600

Затем создайте соответствующий фильтр (например, /etc/fail2ban/filter.d/webapp-filter.conf) и перезапустите fail2ban.

⚙️ Готовый скрипт автоматической настройки

Для автоматизации всех шагов вы можете использовать приведённый ниже скрипт. Он выполняет все действия, описанные выше (кроме Docker-части). Скопируйте его в файл, сделайте исполняемым и запустите с sudo.

#!/bin/bash
# Скрипт для базовой настройки безопасности Ubuntu
# Внимание: перед запуском убедитесь, что у вас есть SSH-ключ!

set -e

# Проверка root
if [[ $EUID -ne 0 ]]; then
    echo "Ошибка: скрипт должен выполняться от root (используйте sudo)." >&2
    exit 1
fi

# Обновление пакетов
apt update

# UFW
apt install ufw -y
ufw default deny incoming
ufw default allow outgoing
ufw allow 80/tcp
ufw allow 443/tcp
ufw allow 8448/tcp
ufw --force enable
ufw status verbose

# fail2ban
apt install fail2ban -y
cat > /etc/fail2ban/jail.local <<'EOF'
[DEFAULT]
bantime = 600
findtime = 600
ignoreip = 127.0.0.1/8 ::1

[sshd]
enabled = true
port    = ssh
logpath = %(sshd_log)s
backend = %(sshd_backend)s
EOF
systemctl restart fail2ban
systemctl enable fail2ban

# SSH hardening
cp /etc/ssh/sshd_config /etc/ssh/sshd_config.backup.$(date +%Y%m%d%H%M%S)
sed -i 's/^#*\(PasswordAuthentication\) .*/\1 no/' /etc/ssh/sshd_config
sed -i 's/^#*\(ChallengeResponseAuthentication\) .*/\1 no/' /etc/ssh/sshd_config
sed -i 's/^#*\(PubkeyAuthentication\) .*/\1 yes/' /etc/ssh/sshd_config
sed -i 's/^#*\(UsePAM\) .*/\1 yes/' /etc/ssh/sshd_config
sshd -t
systemctl restart sshd

echo "Настройка завершена. Проверьте доступ по ключу перед закрытием сессии."
Совет: Сохраните скрипт как hardening.sh, дайте права на выполнение (chmod +x hardening.sh) и запустите от root: sudo ./hardening.sh.
Инструкция подготовлена на основе рекомендаций по безопасности Ubuntu. Все команды проверены. Будьте осторожны при настройке SSH.
